# NWSL App

A personal-project iOS app for **National Women's Soccer League** fans — built to production quality,
used daily, and shipping through TestFlight toward the App Store.

The product is the *fandom*, not the scoreboard. Scores and standings must work — they're table stakes.
The differentiator is everything around them: a team-color design system that paints the entire UI in
your clubs' colors, an AI-powered content feed that balances representation across every team you follow,
interactive Fan Zone games, and a live notification pipeline that puts match moments on your lock screen
the second they happen.

## Architecture — three repos

The app is the client of a deliberately minimal backend split across three repositories:

| Repo | What it does |
|---|---|
| **NWSLApp** *(this repo)* | SwiftUI iOS client — all UI, view models, local state |
| [**nwslapp-proxy**](https://github.com/tiffanyrieth/nwslapp-proxy) | Cloudflare Worker — caches ESPN's unofficial endpoints, enriches them (rosters, weather, crests, social feed), and serves the AI-tagged content pipeline |
| [**nwslapp-match-watcher**](https://github.com/tiffanyrieth/nwslapp-match-watcher) | Cron Cloudflare Worker — polls live matches, drives push notifications + Live Activity updates via APNs |

Per-user state (follows, alert preferences, Fan Zone submissions, leaderboard entries) lives in
**Supabase** (Postgres + Row-Level Security), gated by **Sign in with Apple**.

## Features

- **Follow your clubs** — pick one or many NWSL teams (plus 90+ national teams). Follows persist offline, restore across reinstalls via Supabase sync.
- **Team-color design system** — the entire UI repaints itself in your teams' colors: match cards, standings rows, schedule gradients, notification thumbnails. Dark appearance throughout.
- **Scores · schedule · standings** — full regular season + playoffs, live football-minute clock (`45'+2'`), and a playoff bracket built into the Schedule tab (round-by-round sections with clinch logic — the bracket *is* the schedule).
- **Fan Zone** — four interactive games, each with a distinct color identity:
  - *Predict the XI* — build your predicted lineup on an interactive formation pitch before kickoff
  - *Bracket Battle* — predict the full playoff bracket with themed creative rounds
  - *NWSL Trivia* — daily quiz on league history and current season
  - *Know Her Game* — weekly AI-generated deep-dive quiz on a featured player (see [Engineering highlights](#engineering-highlights))
- **Social feed** — reporters (Bluesky), player Instagram, club social, and news articles aggregated into a filterable feed with per-source toggle controls.
- **Live match alerts** — opt-in pushes for kickoff, goals, halftime, full-time, red cards, lineups, and VAR corrections. Server-rendered notification thumbnails with team crests.
- **Live Activities** — real-time scores on the lock screen and Dynamic Island via push-to-start (iOS 17.2) and broadcast channels (iOS 18).
- **Tips, not ads** — monetized entirely through StoreKit tip jar (Corner Kick → Hat Trick). No ads, no paywalls, no tracking.

## Engineering highlights

### AI-powered content pipeline

The proxy's `/feed` endpoint aggregates content from five source types (Bluesky, Instagram, YouTube, club websites, news RSS). The untrusted firehose — reporter and league-outlet Bluesky posts — is filtered by **Claude Haiku** (`claude-haiku-4-5`), which judges each post for NWSL relevance and tags it with team abbreviations; verdicts are KV-cached by post id (~7 days), so a given post hits the model only once. Club-official and player accounts are **trusted fast-paths** that carry their own team tags and never touch the model. The app receives pre-tagged, normalized JSON and never calls an AI model directly — all inference is server-side.

**Know Her Game** uses Claude the same way: quiz questions are generated by Claude from a **guardrailed prompt** — a **5-layer guardrail system** that prevents gendered framing (never defining a player through a more-famous man), enforces sourcing standards, and validates question quality — then ingested through the proxy's admin route and served to the app. Full spec: [`docs/know-her-game.md`](docs/know-her-game.md).

### Content balancing algorithm

[`ContentRoundRobin.swift`](NWSLApp/Services/ContentRoundRobin.swift) — a pure, deterministic algorithm that guarantees equal screen-time across followed teams regardless of how much content each club produces. A team that posts 50 times a week never crowds out a team that posts 5. Slot allowances scale by follow count, article-priority logic floats news to the top on cold opens, and pull-to-refresh advances each team's window so users always see fresh cards. 20 unit tests.

### End-to-end live notification pipeline

Match event on the pitch → ESPN → proxy → watcher cron → APNs → phone, split across **V1** rich pushes (the interrupt: buzz + server-rendered crest-card thumbnail) and **V2** Live Activity updates (the quiet glance). Fans out at scale via **Cloudflare Queues** and **APNs Broadcast Channels**. Full pipeline: [`docs/notifications.md`](docs/notifications.md).

### Live Activities

iOS 17.2 push-to-start + iOS 18 broadcast channels. A monotonic match clock that works around ESPN's frozen-stoppage-time quirk (the clock freezes at 45:00 and 90:00 during added time — the app reconstructs real time from state transitions). Manual: [`docs/live-activity-v2.md`](docs/live-activity-v2.md).

### Defensive data layer

ESPN's endpoints are unofficial and full of quirks: String-typed scores, a clock that freezes during stoppage time, a full-season query that goes stale mid-match. The app and proxy decode defensively and route around each trap. See [`docs/backend.md`](docs/backend.md).

### No silent failures

Every unexpected condition — fallback path, parse error, stale serve, retry — emits to a `Diagnostics` telemetry spine. The app fails *loud* to the engineer and *honest* to the user. Audit: [`docs/silent-failure-audit.md`](docs/silent-failure-audit.md).

## Stack

| Layer | Tech |
|---|---|
| Language & UI | **Swift 5.9+ / SwiftUI**, `@Observable`, min **iOS 17.2** |
| Architecture | **MVVM** — strict layer split: Models / Services / Stores / ViewModels / Views / Components / DesignSystem |
| Targets | 1 app + 2 extensions (Live Activity widget, Notification Service) + unit & UI tests |
| Networking | **`URLSession` + async/await** — no third-party HTTP library |
| Auth & state | **Supabase** (`supabase-swift` — the *only* third-party dependency) for Sign in with Apple + durable per-user state via Row-Level Security |
| Backend | **Cloudflare Workers** (proxy + match watcher), **Supabase** (Postgres), **Anthropic Claude** (Haiku for feed relevance; a guardrailed prompt for Know Her Game quizzes) |
| Testing | **Swift Testing** (`@Test` / `#expect`) |
| CI | Xcode, TestFlight |

One direct dependency. Everything else is first-party Apple frameworks or the backend Workers.

## Getting started

Requires Xcode 16+ and an iOS 17.2+ simulator or device.

```bash
git clone https://github.com/tiffanyrieth/NWSLApp.git
cd NWSLApp

# Supabase credentials are gitignored — create yours from the template:
cp NWSLApp/Config/Secrets.example NWSLApp/Config/Secrets.swift
# Fill in your Supabase project URL + anon key (Project Settings → API)

open NWSLApp.xcodeproj
```

The anon key is a public client key — **Row-Level Security** is the real auth boundary, so it's
excluded from git as hygiene, not secrecy. Live data flows through the proxy and watcher Workers
(their own repos); without them the app degrades to honest "couldn't load" states.
